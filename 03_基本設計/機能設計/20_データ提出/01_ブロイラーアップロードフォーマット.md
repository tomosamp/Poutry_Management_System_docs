# ブロイラーアップロードフォーマット機能設計

## CSVインポート前提
- ソースファイルはGoogleフォームからエクスポートされたCSV（カンマ区切り、ダブルクォート囲み有無混在、1行目がヘッダー）を想定し、UTF-8（BOM有無どちらでも可）でアップロードする。
- ファイルはユーザー側で加工せずにそのまま取り込まれる前提とし、システム側でエラーを早期検知・通知する。
- 1ファイルあたり最大行数はフォーム出力の実績（約70行）に余裕を持たせて5,000行までを上限とする。超過時はファイル単位でエラーとする。

### 自動正規化ポリシー
システム側で安全に実施できる最小限の正規化のみ許可する。
- 先頭末尾の半角・全角スペース、タブの除去。
- 全角英数字・句読点を半角へ統一（例: `０７９６７２５１１１` → `0796725111`）。
- 数値項目（No.12〜26）の3桁区切りカンマ除去、および全角カンマ→半角カンマ変換。
- 数値項目（No.12〜26）は全角数字を半角に変換し、正規化後に半角数字以外の文字が残った場合は値を空（NULL登録）として扱う。
- 改行コードはCRLF・CRをLFに統一。
上記以外（記号削除、語句置換など）は行わず、該当する値はバリデーションエラーとする。

## 処理フロー概要
1. ユーザーが管理画面からCSVを選択しアップロード。
2. ファイルレベル検証（サイズ・行数・文字コード・列数・1行目ヘッダー有無）。
3. 自動正規化を実施後、列位置（A〜AB）でシステム内部カラムへマッピングし、数値列のクレンジング（半角化・カンマ除去・非数値のNULL化）を行う。
   - ヘッダー行の文字列は**検証せず**、列位置のみを使用する（ヘッダー名は運用上の確認用に本書へ記載）。
   - 重量はDBにkg（整数）で保持する（`live_processed_weight_kg`）。
4. レコード単位のバリデーションを実施。行ごとにエラーを蓄積。
5. エラーが1件でも存在する場合は登録処理を打ち切り、エラー一覧（行番号・項目・理由）を利用者へ返却。NULL化された項目は警告リストにまとめ、取込後ダウンロードできるようにする。
6. エラーがなければデータストアへ登録し、取り込み件数およびNULL化件数をレスポンスする。

## バリデーションおよびエラー判定一覧

### ファイルレベル
- **文字コード不一致**：UTF-8以外（Shift-JIS等）のファイルはエラー。
- **ヘッダー欠落／余分な列**：想定28列（A〜AB）と一致しない場合。C列のスコアは固定で存在する前提だが、システムでは利用しない。
- **空ファイル／データ行なし**：ヘッダーのみ、あるいは完全空の場合。
- **CSV構造破損**：クォート未閉じや列分割崩れ（例: コメント欄内の改行が未クォート）を検出した場合。

### ヘッダー定義
期待するヘッダー名（順序固定、文字列は検証しない）は以下の通り。原票で改行を含む設問文は正規化後に1セルへ格納されている前提とする。
1. タイムスタンプ
2. メールアドレス
3. スコア（Googleフォーム独自。システムでは利用しない）
4. a【北海道・東北地区】
5. b【関東地区】
6. c【中部地区】
7. d【近畿・中国・四国地区】
8. e【北部九州地区】
9. f【南部九州地区】
10. 「部署名」をご記入ください（取り込み対象外）
11. 「お名前」をご記入ください（取り込み対象外）
12. 「お電話番号」をご記入ください（取り込み対象外）
13. ひな入雛羽数（実績・提出月-1）
14. ひな入雛羽数（予定・提出月）
15. ひな入雛羽数（予定・提出月+1）
16. ひな入雛羽数（予定・提出月+2）
17. 生体処理羽数（実績・提出月-1）
18. 生体処理羽数（予定・提出月）
19. 生体処理羽数（予定・提出月+1）
20. 生体処理羽数（予定・提出月+2）
21. 生体処理重量（実績・提出月-1）
22. 生体処理重量（予定・提出月）
23. 生体処理重量（予定・提出月+1）
24. 生体処理重量（予定・提出月+2）
25. 禁止（羽）
26. 全部廃棄（羽）
27. 一部廃棄（羽）
28. コメント（CSV備考）

### DBマッピング（主要）
- 登録先: `monthly_records`
- CSVメタ情報:
  - `input_timestamp`: A（タイムスタンプ）
  - `source_email_raw`: B（メールアドレス。受領文字列をそのまま保持）
  - `note_text`: AB（コメント）
- 対象月の扱い（提出月=取込画面で選択した年月）:
  - 実績: 提出月の先月を `data_kind=actual` として登録
  - 予定: 提出月〜+2ヶ月を `data_kind=plan` として登録（計3ヶ月）

### レコード共通チェック
- 行内のカラム数が28と一致しない場合（カンマ不足・超過）は即時エラー。
- すべてのセルが空（正規化後）な行はスキップ対象とし、1件でも値があればバリデーションを実施。
- 1行あたりの文字数は10,000文字を上限とする。超過時はコメント欄過多とみなしエラー。
- 改行を含むセルはクォートで囲まれている必要がある。未クォートの改行は構造破損として扱う。

### 項目別チェック

#### 地区選択列（a〜f）
- 6列のうち、値が入るのは1列のみ。複数列へ入力されている場合はエラー。
- 値は「`a3｜日本ホワイトファーム㈱ 東北食品工場`」のように、地区コード＋全角縦線＋事業所名の形式であること。
- 非対応のコード（例: `g`系、空欄、自由入力）はエラー。
- 完全空欄の場合は地区未選択扱いでエラー（地区必須）。
- **生産者（施設）の割り出し**: D〜I のうち最初に値が入っているセルを採用し、`｜` より前（例: `a3`）を `facility_import_key`（小文字）として抽出する。
  - 例: `a3｜日本ホワイトファーム㈱ 東北食品工場` → `facility_import_key=a3`
  - 照合: `facilities` を鳥種（ブロイラー）で絞り、`facilities.import_key` と完全一致で特定する（名称の曖昧一致は行わない）。
  - 施設名（`｜` 以降）は表示・ログ用途のため保持するが、照合には使用しない。

#### 基本属性列
- **タイムスタンプ**：`M/D/YYYY H:MM:SS` または `YYYY/MM/DD HH:MM:SS`（24時間表記）で解析できること。不存在日付・未来日付（当日より2日以上未来）はエラー。
- **メールアドレス**：受領した文字列をそのまま `source_email_raw` に保持する（正規化しない）。メール形式は厳格にエラーにせず、必要に応じて警告とする。
- **「部署名／お名前／お電話番号」**：取り込み対象外（値は保持せず、バリデーション対象外）。

#### 羽数列（M〜T）
- 対象列：No.13〜20。
- M〜P はひな入雛羽数（実績＋予定3ヶ月）、Q〜T は生体処理羽数（実績＋予定3ヶ月）。
- 正規化後に半角数字のみで構成されている場合は整数として取り込む。
- `-` / `なし` / `不明` / `記録なし` / `データなし` は「データなし」とみなし **0として登録**（警告）。
- `18羽` など単位付きの場合は数値を抽出して登録（警告）。
- 9桁を超える場合は上限超過としてエラー（異常値）とみなす。

#### 重量列（U〜X）
- 対象列：No.21〜24。
- 小数を許容せず、半角数字のみで構成されている場合に限り整数として取り込み（単位はKg）。
- それ以外（小数点・文字列混在・単位付き等）の場合はデータなし扱い（0登録）。**重量の小数点以下は警告なし**。
- 9桁を超える場合は上限超過としてエラー。

#### 廃棄関連列
- 対象列：No.25〜27。
- 値が半角数字のみで構成されている場合に整数として取り込み。
- `-` / `なし` / `不明` / `記録なし` / `データなし` は「データなし」とみなし **0として登録**（警告）。
- 単位付きは数値を抽出して登録（警告）。
- マイナス値はエラーとして扱う。

#### コメント列
- 任意入力。最大1,000文字まで許容（システム側でカウント）。超過時はエラー。
- 改行は保持するが、HTMLタグや制御文字はエラー。
 - 取込時は `monthly_records.note_text` に保存する。

### エラー・警告メッセージ方針
- エラーは「行番号」「列名」「発生理由」「該当値」の4要素で表示する。
- 数値列の「データなし」補正・単位除去は警告として別途一覧を返す。
- 同一行に複数エラーがある場合はすべて列挙し、ユーザーが一度に修正できるようにする。
- 代表的な文例  
  - `行23: メールアドレス → メール形式が不正の可能性があるため警告にしました（入力値: '氏名' <xxx@yyy>）。`
  - `行41: １１月予定数値（単位：羽） → データなしのため0として登録します（入力値: 不明）。`
  - `行52: a【北海道・東北地区】 → 地区列は1列のみ入力してください（a,bの複数入力を検出）。`

### 取込結果通知
- 正常終了時は登録件数・スキップ件数・NULL化件数を画面上に表示し、CSVの原票保管先（オブジェクトストレージ等）の参照リンクをログ出力する。
- 警告が存在する場合は、NULL化・補正内容をまとめた警告CSVをダウンロード可能とする。
- エラー終了時は処理をロールバックし、エラー一覧をCSVファイルとしてダウンロードできるようにする（同じ列構成＋エラー理由列）。

### 留意・懸念事項
- Kg列に小数値（例: 2,174,587.7）が入力されている場合、今回の仕様では整数化できないためNULL化される。業務側が小数情報を引き続き必要とする場合は別途入力ルールの周知が必要。
- 重量の小数点以下によるNULL化は運用上許容し、警告は出さない（他項目のNULL化は必要に応じて警告対象とする）。
- 「部署名」を任意としたことで空欄の提出が増える可能性がある。レポート表示で部署名が必須な画面がある場合は別途対応が必要。
- メールアドレスは生保持のため、形式揺れは警告で通知する。フォーム側での入力ガイドやサンプル提示を検討。
